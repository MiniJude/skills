# 技术写作风格

使用这个 skill 时，要写出用户偏好的技术文章：实战、鲜活、有证据、重过程，并且让读者看完之后真的知道怎么做。

## 核心定位

- 写成技术实践文章，不写成某个业务系统的宣传或项目介绍。
- 业务领域通常不重要。除非用户明确要求，否则不要围绕 CRM、ERP、行业、客户、内部项目背景展开。
- 标题和主线要强调最终效果，而不是强调中间工具。
- 仓库结构、工具、平台、框架名都只是手段，不是文章主角。
- 钉钉、飞书、企微、Slack、GitHub、Gitee、阿里云、腾讯云等平台名可以在正文作为例子出现，但不要主导标题，除非文章就是专门讲这个平台。

## 标题风格

标题要优先满足这些特点：

- 效果优先：先告诉读者最终能得到什么，而不是先讲用了什么技术。
- 有钩子：可以有一点戏剧性、口语感、趣味性，但不能空。
- 具体：最好能让读者看到一个明确结果或场景。
- 适合系列化：主标题统一，副标题区分每篇重点。
- 合适时可以使用 emoji，增强传播感。

推荐标题模式：

- `你还在古法编程吗？🦞 把小龙虾接进工作群（一）：先让 AI 看懂前后端`
- `背百遍八股，不如一次实操：从带宽打满到首屏优化`
- `两月没动的项目突然爆红：一场由 ESLint 配置缺失引发的血案`
- `从买服务器到搞定一切：帮朋友搭建企业技术基建（一）`
- `我踩了大家都会踩的坑：一次小程序开发复盘`

避免过于“实现细节中心”的标题：

- `把代码仓库变成群聊可开发系统`
- `Git Subtree 合并前后端仓库实践`
- `OpenClaw 接入钉钉机器人教程`

如果一个标题准确但太像内部技术说明，要改成结果导向：

- `前后端合仓实践` -> `先让 AI 看懂前后端`
- `发布回滚脚本设计` -> `让它敢发测试、能回滚`
- `OpenClaw 接入工作群` -> `聊聊天就把活干完`

## 开头写法

文章开头先讲痛点和目的。

前几段要回答：

- 这篇文章由什么真实问题触发？
- 读者为什么要关心？
- 看完之后能做到什么？
- 哪些背景其实不重要？

常用开头方式：

- `问题不在于项目属于哪种产品形态。具体领域不重要，真正的问题是...`
- `背百遍八股不如一次实操。`
- `今天朋友请我帮忙搞一下...在此记录全过程。`
- `过去两个月没碰这个项目，再次打开突然一片红。`
- `这不是为了追求某个概念，而是为了解决一个真实协作问题。`
- `今天部署好某工具后发现核心功能没了，于是我开始找平替。`
- `这篇文章讨论的不是某个列表怎么做，而是某种体验应该如何落地。`
- `首先，要明确几个原则。`

除非文章就是概念解释，否则不要一上来就写抽象定义。即使要讲理论，也先用真实现象或真实事故把读者拉进来。

## 文章结构

优先使用清晰的编号标题：

```text
一、问题现象
二、原因分析
三、方案选择
3.1 方案 A
3.2 方案 B
四、实操过程
五、验证效果
六、截图建议 / 注意事项
七、小结
```

实操型文章推荐流程：

1. 先讲痛点。
2. 再讲目标。
3. 展示初始状态。
4. 对比可选方案。
5. 一步步执行。
6. 解释关键命令或代码。
7. 用可见证据验证结果。
8. 总结可复用的方法。

基础设施教程推荐流程：

1. 先说明最终目标，例如“公网可访问”“HTTPS 可用”“对象存储可管理”。
2. 说明当前环境和前置条件。
3. 每一步都给出命令。
4. 每一步都说明如何验证。
5. 标出容易踩坑的配置项，例如编码、端口、安全组、路径、权限、续期。
6. 最后给出长期维护建议，例如自动续期、日志位置、备份、权限控制。

排查型文章推荐流程：

1. 先贴问题现象或错误信息。
2. 说明第一个猜测为什么合理。
3. 说明如何验证或推翻这个猜测。
4. 顺着下一个线索继续排查。
5. 找到根因。
6. 给出最小修复。
7. 增加防止再次发生的约束或规范。

体验问题技术化文章推荐流程：

1. 先定义体验目标，而不是直接写实现。
2. 把用户会遇到的不合理体验拆成具体问题。
3. 给出判断标准，例如“不重复打扰”“不遗漏”“刷新后继续”“跨标签页接力”。
4. 再把判断标准映射到技术方案。
5. 用边界场景验证方案，例如多标签页、刷新、弱网、重复触发、并发请求。

AI 自动化流程文章推荐流程：

1. 先说理想闭环是什么。
2. 再明确原则，例如上下文窗口拆分、Skill 只解决一类流程、MCP 与 Skill 分工。
3. 把流程拆成可被 AI 执行的步骤。
4. 标出哪些动作可以自动化，哪些动作必须人工确认。
5. 强调日志、审计、权限和失败兜底。

## 语气风格

- 可以使用第一人称实操叙述：`我发现`、`我猜测`、`我验证`、`我最后选择`。
- 可以保留口语化、有记忆点的表达：`古法编程`、`擦屁屁`、`血案`、`踩坑`、`爆红`。
- 口语感必须服务于技术内容，不能变成纯营销文案。
- 段落要短，标题要清楚。
- 文章要像真实工程师写的复盘，而不是泛泛的教程拼贴。

## 证据和截图

用户过往文章常用“可见证据”推进：Network 面板、依赖分析图、终端输出、配置文件、性能数据、前后对比截图、发布日志等。

写文章时，合适的位置要补“截图建议”：

- 初始问题截图。
- 修改前的目录或工具状态。
- 命令执行结果。
- 浏览器 devtools / network / timing 面板。
- 可视化分析报告。
- 配置修改前后对比。
- 发布日志或审计记录。
- 最终效果截图。

截图必须提醒打码：

- `.env`
- token
- 数据库账号密码
- 服务器 IP
- 手机号
- 私有域名
- 生产环境路径

## 技术深度

文章要同时讲“怎么做”和“为什么这么做”。

不要只罗列命令，要解释：

- 这一步为什么需要。
- 它证明或排除了什么问题。
- 接受了什么取舍。
- 可能的失败点是什么。
- 如何验证结果。

代码块要可复制，但不要大段堆文件。优先使用短小、聚焦的命令和片段。

常用解释方式：

- `这个配置的作用是...`
- `如果只做到这里，会有一个问题...`
- `为了验证这个猜测，我...`
- `最终业务代码几乎只需要改...`
- `这里不选择方案 B，是因为...`

## 通用性规则

当用户要写公开技术博客时：

- 项目名尽量泛化为 `项目`、`前端`、`后端`、`web-admin`、`web-mobile`、`backend/api`、`服务`、`工作群`。
- 文章要能被不同团队复用。
- 只有当工具对技术方案本身很关键时，才把工具名写进主线。
- 如果工具只是载体，就放到正文作为例子，不放在标题中心。

示例：

- 推荐：`工作群可以是钉钉、飞书、企业微信或其他 IM 工具，平台不重要，重要的是权限、确认和审计。`
- 避免：`钉钉机器人实战` 作为系列总标题，除非文章专门讲钉钉。

## 特殊题材写法

### 热点事件 + 平替实操

当文章由工具变更、商业限制、开源争议、功能移除等事件触发时，可以立场鲜明，但要快速回到实操：

- 先讲发生了什么。
- 再讲为什么这影响真实开发。
- 然后找替代方案。
- 接着给出部署、配置、验证过程。
- 最后说明这个替代方案的边界和风险。

这种写法适合类似“某工具社区版移除核心功能，我来试试平替”的文章。

### 手把手基础设施

当文章是证书、服务器、Nginx、对象存储、域名、CI/CD 等基础设施教程时，要把读者当成“跟着做的人”：

- 命令必须完整。
- 前置条件必须明确。
- 每一步都要有验证方式。
- 遇到平台差异时要说明，例如 apt / yum / dnf、Ubuntu / CentOS / Alibaba Cloud Linux。
- 重点提醒长期维护动作，例如证书续期、服务开机自启、日志路径、备份。

### 体验问题技术化

当文章来自产品体验问题时，不要只讲代码。先把体验目标说清楚：

- 哪种体验是不合理的？
- 用户为什么会被打扰？
- 哪些场景不能漏？
- 方案如何同时满足“不重复”和“不遗漏”？

再进入实现细节。

### AI 流程自动化

当文章讲 AI、Skill、MCP、Agent、ChatOps 时，要先讲流程原则：

- 一个上下文窗口尽量只干一类相关任务。
- Skill 尽量拆分，一个 Skill 专注解决一种流程。
- MCP 更像工具调用，Skill 更像流程编排。
- 高风险动作要有人确认。
- 自动化一定要有日志和审计。

不要把 AI 写成万能助手，要写成“有边界、可验证、可回滚的流程执行者”。

## 修改文章时的规则

当用户基于文章提出反馈时：

- 把最新的写作偏好当成长期偏好。
- 如果反馈具有复用价值，要更新这个 skill。
- 优先删掉啰嗦、宽泛、离题的背景。
- 把长段落改成有标题的步骤。
- 把标题改成更强调效果。
- 实现细节保留在正文，不要抢标题主角。
- 如果用户要求保留参考链接，必须保留。

## 从参考文章中提炼出的风格

可复用特征：

- 多数文章来自真实实操：优化、调试、部署、集成、基础设施搭建、线上问题复盘。
- 喜欢清晰编号结构，常用 `一、二、三` 和 `1.1 / 2.1`。
- 开头通常先讲真实问题、真实事故或真实目标。
- 过程里会出现截图、命令、配置、数据、Network 面板、性能指标等证据。
- 喜欢先比较方案，再说明为什么选当前方案。
- 语气可以口语化、有戏剧性，但会回到技术细节。
- 高频主题包括：`实测`、`踩坑`、`复盘`、`总结`、`帮朋友`、`问题分析`、`解决方案`。
- 目标不是单纯解释概念，而是让读者能复现结果。
- 新补充的 4 篇进一步体现出这些倾向：
  - 能从热点或争议切入，但最终落到替代方案和部署实操。
  - 基础设施文章偏手把手，强调前置条件、命令、验证和长期维护。
  - 产品体验类问题会先提出判断标准，再映射到技术方案。
  - AI 自动化文章会先讲原则和流程边界，再讲工具组合。
