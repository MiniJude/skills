# Frontend Design Preferences

本 skill 记录用户跨项目复用的前端页面展示和交互偏好。项目特有的产品语义、端口、账号、领域模型和历史约定应留在具体项目自己的 skill 中。

## 优先级

- 当前对话中的具体要求优先。
- 如果本 skill 与项目已有设计系统冲突，先尽量在现有设计系统内实现；无法兼容时说明冲突点并确认。
- 新增或修改 UI 时，优先复用项目已有 token、class、组件变体和交互样式。

## 输入框

- 所有可编辑输入框都应提供一键清空 icon。
- 一键清空 icon 只在有内容且输入框可编辑时出现。
- 清空后应保持当前操作流顺畅，通常保留焦点在输入框内。
- 搜索框、筛选框、表单文本框、多行输入框、命令输入框等都适用。

## 主题与视觉一致性

- 同一产品内的页面、组件、弹层和状态应保持统一的颜色、圆角、阴影、间距、字体层级和交互动效。
- 如果页面之间存在历史风格差异，新增实现应贴近当前主线风格，并逐步收敛差异。
- 引入第三方组件或新 UI 模式时，要确认它能被当前主题体系统一接管。

## 移动端与平板适配

- 前端页面必须适配移动端和平板，不应只按桌面宽度设计。
- 对话类或多栏应用在移动端应优先展示当前内容，左侧菜单默认收起。
- 移动端/平板宽度下，页面标题或工具栏应提供菜单 icon，用于展开左侧菜单。
- 左侧菜单展开时应以抽屉形式覆盖大部分屏幕宽度，右侧主内容保留一部分可见，并使用蒙层压暗；点击蒙层可关闭菜单。
- 移动端抽屉和蒙层不能导致页面横向滚动，也不能破坏当前内容区滚动位置。

## 交互状态一致性

- 同一界面中语义、层级和交互方式相近的控件，必须保持一致的视觉反馈。
- 图标按钮、展开/收起、编辑、更多、新建、刷新等普通操作应使用统一的中性 hover/active/focus/selected/disabled 状态。
- 删除、归档等危险操作可以使用危险态，但危险态也要在同一规则体系内。
- 不要为单个 icon 临时写独立 hover 规则，除非它有明确语义差异。

## Loading 状态

- 页面主体、大面板、列表、表格、详情区等大面积内容加载时，应优先使用骨架屏，不要只显示居中的转圈和“正在加载”文案。
- 由用户点击后展开的完整面板也属于大面板 loading，例如详情面板、文件列表面板、抽屉/侧栏内容区、弹层主体内容区。
- 骨架屏应复用现有主题变量和项目已有 loading 质感，并尽量模拟真实内容结构，避免加载完成前后布局跳动。
- 小范围操作反馈可以继续使用转圈，例如按钮内、icon 按钮内、局部刷新图标、单个行内操作等。

## Flex 填充与滚动

- 后台、工具页、三栏布局、详情面板、列表面板等需要吃满可用空间的区域，应优先使用主流 flex 填充方案。
- Tailwind 项目中优先使用 `flex flex-col min-h-0`，主要内容使用 `flex-1 min-h-0 overflow-y-auto`。
- 不要用随意的死高度或 `calc(...)` 硬撑主要面板高度，除非组件语义上确实需要固定尺寸。
- 多层 flex/grid 嵌套时，凡是内部需要滚动的父级都要检查 `min-height: 0` / `min-width: 0`。

## Icon 与提示

- 单独展示、没有文字标签解释的 icon 必须提供 tooltip 或 popover 提示。
- icon 提示文案应直接说明操作含义，例如“清空”“删除”“归档”“新建”“刷新”。
- 图标按钮应保持可访问名称，例如 `aria-label`、`title`、tooltip/popover 内容或组件库等价能力。
- Tooltip、Popover、Popconfirm 等浮层不能超出可视区域，也不能撑开页面或容器导致新滚动条。
- 浮层应根据触发点和视口边界自动调整位置，必要时从上方/下方切换展示，并对左右边界做夹取。

## 关键操作确认

- 删除、归档、清空大量内容、重置配置、断开连接、覆盖保存、批量修改等关键或不可轻易撤销的操作，需要用户二次确认。
- 二次确认应使用靠近触发点的 Popconfirm，不使用全局弹框确认。
- Popconfirm 文案应明确说明操作对象和后果，避免只写“确定吗”。
- 高风险操作的确认按钮应使用危险态或强调态，取消按钮应清晰可见。
- 如果操作可以撤销，也可以搭配 undo/snackbar，但不能因此省略关键操作的必要确认，除非用户明确要求。
